# 장애 대응 & 포스트모템

상황: $ARGUMENTS

## 모드 1: 장애 대응 가이드

### 심각도 분류
- **P0 (Critical)**: 서비스 전체 중단, 데이터 유실, 보안 침해
- **P1 (High)**: 핵심 기능 장애, 성능 심각 저하
- **P2 (Medium)**: 일부 기능 장애, 우회 가능
- **P3 (Low)**: 경미한 이슈, 사용자 영향 미미

### 대응 체크리스트
```
□ 1. 심각도 분류 및 담당자 지정
□ 2. 영향 범위 파악 (사용자 수, 기능, 데이터)
□ 3. 커뮤니케이션 시작 (Slack/상태 페이지 업데이트)
□ 4. 원인 조사 (로그, 메트릭, 최근 배포)
□ 5. 완화 조치 (롤백, 스케일링, 기능 플래그)
□ 6. 모니터링 (복구 확인)
□ 7. 상태 업데이트 (해결 선언)
□ 8. 포스트모템 일정 잡기
```

### 빠른 진단 순서
1. **최근 배포가 있었는가?** → 롤백 고려
2. **인프라 이슈인가?** → 클라우드 상태 페이지 확인
3. **트래픽 급증인가?** → 스케일링/캐싱
4. **외부 서비스 장애인가?** → 의존 서비스 상태 확인
5. **데이터 이슈인가?** → DB 상태, 마이그레이션 확인

## 모드 2: 포스트모템 작성

### 문서 구조

```markdown
# 포스트모템: [장애 제목]

**날짜**: YYYY-MM-DD
**심각도**: P0/P1/P2/P3
**작성자**: [이름]
**상태**: 초안 / 리뷰 완료 / 액션아이템 진행 중

## 요약
[장애를 2-3문장으로 요약. 무엇이, 언제, 얼마나 영향]

## 영향도
- **영향 시간**: HH:MM ~ HH:MM (총 X시간)
- **영향 사용자**: 약 N명
- **영향 범위**: [어떤 기능/서비스]
- **데이터 손실**: 있음/없음
- **금전적 영향**: (해당 시)

## 타임라인
| 시각 | 이벤트 |
|------|--------|
| 10:00 | 배포 시작 |
| 10:15 | 에러율 증가 감지 |
| 10:20 | 알림 발생, 담당자 확인 시작 |
| 10:35 | 원인 파악 (DB 커넥션 풀 고갈) |
| 10:40 | 롤백 실행 |
| 10:50 | 서비스 복구 확인 |

## 근본 원인 분석 (5 Whys)
1. **왜** 서비스가 중단되었나? → DB 커넥션 풀 고갈
2. **왜** 커넥션이 고갈되었나? → 새 쿼리가 커넥션을 반환하지 않음
3. **왜** 반환하지 않았나? → 트랜잭션 타임아웃 미설정
4. **왜** 타임아웃이 없었나? → 코드 리뷰에서 누락
5. **왜** 누락되었나? → DB 연동 체크리스트 부재

**근본 원인**: DB 트랜잭션 타임아웃 미설정 및 코드 리뷰 체크리스트 부재

## 완화 조치 (당시 적용)
1. 배포 롤백
2. DB 커넥션 풀 수동 초기화

## 액션 아이템
| 우선순위 | 항목 | 담당자 | 기한 | 상태 |
|---------|------|--------|------|------|
| P0 | 트랜잭션 타임아웃 설정 추가 | - | - | 대기 |
| P1 | DB 커넥션 풀 모니터링 알림 추가 | - | - | 대기 |
| P2 | DB 연동 코드 리뷰 체크리스트 작성 | - | - | 대기 |

## 교훈
### 잘한 점
- 빠른 감지 및 롤백 결정

### 개선할 점
- 모니터링 알림 임계값 재조정 필요
- 배포 전 부하 테스트 추가

## 참고 자료
- [관련 Slack 스레드 링크]
- [대시보드 스크린샷]
```

## 규칙
- 비난하지 않는 문화 (blameless) — 사람이 아닌 시스템에 초점
- 타임라인은 구체적 시각 포함
- 액션 아이템은 측정 가능하고 담당자/기한 명시
- 감정적 표현 배제, 사실 기반 기술
