# 테스트 생성

대상: $ARGUMENTS

## 테스트 작성 프로세스

### 1. 대상 코드 분석
- 함수/메서드의 입출력 파악
- 의존성 확인 (외부 API, DB 등)
- 기존 테스트 패턴 확인 (프로젝트 컨벤션 따르기)

### 2. 테스트 케이스 설계

#### 필수 케이스
- **Happy Path**: 정상적인 입력으로 기대 결과 확인
- **Edge Cases**: 경계값, 빈 입력, 최대값 등
- **Error Cases**: 잘못된 입력, 예외 상황

#### 선택 케이스
- **Integration**: 모듈 간 연동
- **Performance**: 대량 데이터 처리

### 3. 테스트 코드 작성

#### 컨벤션
- 프로젝트의 기존 테스트 프레임워크와 패턴을 따름
- 테스트 파일 위치: 기존 테스트 파일 위치 패턴을 확인
- Arrange-Act-Assert (AAA) 패턴 사용
- 테스트 이름은 `should [expected behavior] when [condition]` 형식

#### 모킹 전략
- 외부 의존성만 모킹 (DB, API, 파일시스템)
- 내부 함수는 가능하면 실제로 실행
- 모킹 라이브러리는 프로젝트에서 사용하는 것을 따름

## 출력 형식

```
## 테스트 계획

**대상**: [함수/모듈 이름]
**프레임워크**: [Jest/Vitest/pytest 등]
**케이스 수**: N개

### 테스트 케이스
1. [케이스 설명]
2. ...
```

이후 실제 테스트 코드를 작성합니다.

## 규칙
- 프로젝트의 기존 테스트 스타일을 먼저 확인하고 따르기
- 구현 세부사항이 아닌 동작(behavior)을 테스트
- 하나의 테스트는 하나의 동작만 검증
- 테스트 간 독립성 유지 (순서 의존 금지)
